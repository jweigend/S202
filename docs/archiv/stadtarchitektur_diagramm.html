1. Einleitung & Motivation

Klassische Softwarearchitektur-Diagramme stellen Dependencies als Linien mit Pfeilspitzen dar. Diese Konvention hat fundamentale Schwächen: Linien haben keine Breite, sie transportieren keine Mengeninformation, und bei vielen Abhängigkeiten entsteht ein unleserliches Geflecht.

Das Stadtarchitektur-Diagramm ersetzt diese Konvention durch eine Raummetapher aus dem Städtebau.

Grundprinzip

Je mehr Klassen eine Abhängigkeit zu einem Ziel teilen, desto breiter wird die Straße, die dorthin führt. Breite = Bündelung. Die Stadt zeigt auf einen Blick, welche Strukturen stark gekoppelt sind.

Die vertikale Achse ist bedeutungstragend: weiter oben liegende Elemente hängen ab von weiter unten liegenden. Oben sieht man UI-Klassen, unten Domain und Querschnitt.

2. Die Metaphern

2.1 Pakete = Stadtteile

Ein Java-Paket wird als Stadtteil dargestellt: eine gefüllte Rechteckfläche mit Label oben links. Stadtteile sind keine Barrieren — Straßen laufen durch sie hindurch. Verschachtelte Pakete sind Stadtteile innerhalb von Stadtteilen (dunklere Tönung).

2.2 Klassen = Gebäude

Jede Klasse ist ein Gebäude: ein massives, flächiges Rechteck mit dem Klassenname. Gebäude haben deutlich mehr visuelles Gewicht als in klassischen Diagrammen — sie sind die Hauptakteure, keine bloßen Labels.

2.3 Dependencies = Straßen

Eine Dependency wird als gefüllter Rechteckstreifen dargestellt — nicht als Linie. Die Straße beginnt an der Gebäude-Unterkante der Quellklasse, läuft durch den Korridor zwischen den Stadtteilen, und endet an der Gebäude-Oberkante der Zielklasse. Paketgrenzen unterbrechen die Straße nicht.

Kernregel Straßen enden am Gebäude, nicht am Paket. Eine Paketgrenze ist kein Haltepunkt — die Straße läuft durch den Stadtteil hindurch.

2.4 Korridore = Straßenraum zwischen Stadtteilen

Zwischen je zwei vertikal benachbarten Stadtteilen liegt ein dedizierter Korridor — ein freier Streifen ohne Gebäude (40–60px). In diesem Korridor verlaufen alle Straßen horizontal, bevor sie vertikal in den nächsten Stadtteil einbiegen. Der Korridor ist der Ort der Bündelung.

3. Layout-Regeln

3.1 Vertikale Ordnung

Die vertikale Position ist bedeutungstragend: oben = abhängig, unten = fundamental. Diese Regel gilt sowohl zwischen Stadtteilen als auch innerhalb eines Stadtteils — abhängige Klassen stehen tiefer als ihre Quellen.

3.2 Straßenbreite und Spurprinzip

Jede einzelne Dependency belegt eine Fahrspur fixer Breite (8–10px). Wo mehrere Dependencies dieselbe Strecke nutzen, werden die Fahrspuren nebeneinander geführt und durch gestrichelte weiße Linien getrennt. Spurmarkierungen werden nur gezeichnet wenn n > 1.

4. Routing-Algorithmus

4.1 Normale Straße

1. Vertikal:    Quell-Gebäude Unterkante  →  Korridor-Mitte
2. Horizontal:  Quell-cx                  →  Ziel-cx
3. Vertikal:    Korridor-Mitte            →  Ziel-Gebäude Oberkante

4.2 Split-Ausgang

Wenn ein Gebäude mehrere Dependencies in denselben Korridor entsendet, verlassen sie das Gebäude als gemeinsamer Ausgangsblock mit Spurtrennern und splitten sich erst auf Korridor-Höhe auf — auf exakt derselben Y-Position.

4.3 Bündelung und Merge

Stichstraßen münden im Korridor von rechts in die gebündelte Hauptstraße. Spurmarkierungen beginnen erst am Merge-Punkt — nicht früher.

5. Fast Lanes

Wenn eine Dependency Stadtteile überspringt (z.B. UI direkt zu Domain), nutzt sie eine Fast Lane: einen vertikalen Streifen außerhalb der Stadtteile, der alle übersprungenen Ebenen ungehindert durchläuft.

Fast Lane entsteht organisch

Jede Langstrecken-Dependency belegt eine Fahrspur in der Fast Lane. Die Fast Lane wird breiter, je mehr solche Dependencies sie passieren. Wo keine Langstrecken existieren, gibt es keine Fast Lane.

Route einer Fast Lane:

1. Ausfahrt:   Horizontal aus Gebäude-Mitte → Fast Lane Rand (außen)
2. Fast Lane:  Vertikal außerhalb aller Stadtteile bis zur Einfahrt-Y
3. Einfahrt:   Horizontal → Zielgebäude (zentriert auf dessen cx)
4. Ziel:       Vertikal in Gebäude-Oberkante

5.1 Einmündungs-Reihenfolge

Mehrere Fast-Lane-Spuren biegen auf gestaffelten Y-Positionen ab. Weiter links liegende Ziele biegen früher (höher) ab, weiter rechts liegende tiefer.

7. Implementierungshinweise für JavaFX

7.1 Datenmodell

record Package(String fqn, List<Package> children, List<ClassNode> classes) {}
record ClassNode(String name, Package pkg, List<Dependency> outgoing) {}
record Dependency(ClassNode source, ClassNode target) {}

record PackageLayout(Package pkg, double x, double y, double w, double h) {}
record ClassLayout(ClassNode cls, double cx, double top, double w, double h) {}
record RoadSegment(double x, double y, double w, double h,
                   int laneCount, boolean isVertical) {}
record LaneMarker(double x1, double y1, double x2, double y2) {}

7.2 Layout-Pipeline

  1. Ebenen bestimmen — topologische Sortierung auf Paketebene. Ebene 0 = keine ausgehenden Dependencies.
  2. Gebäude-Platzierung — Klassen horizontal innerhalb eines Pakets; abhängige tiefer stellen.
  3. Korridor-Berechnung — Höhe = max. Spuranzahl × Spurbreite + Padding.
  4. Fast Lane Analyse — Langstrecken identifizieren, rechts/links zuweisen.
  5. Straßensegment-Generierung — Pfad berechnen, Segmente bündeln, Breite zuweisen.
  6. Rendering — Segmente als Rectangle, Spurmarkierungen als gestrichelte Line.

7.3 Z-Order

pane.getChildren().addAll(

7.5 Bündelungsalgorithmus

Map<CorridorSegment, List<Dependency>> bundleMap = new HashMap<>();
for (Dependency dep : allDeps) {
    for (CorridorSegment seg : computeRoute(dep))
        bundleMap.computeIfAbsent(seg, k -> new ArrayList<>()).add(dep);
}

8. Ausblick: Log-Player

Das Stadtarchitektur-Diagramm ist nicht nur ein statisches Dokument — es ist ein Animationsmedium. Die Straßen sind Bahnen, auf denen sich Laufzeit-Ereignisse als Kugeln bewegen.

Grundidee

Ein Log-Player liest Method-Calls oder Traces und spielt diese im Zeitraffer ab. Jeder Call erscheint als Kugel auf der entsprechenden Straße und bewegt sich von der Quell- zur Zielklasse. Farbe, Größe und Geschwindigkeit kodieren Typ, Häufigkeit und Dauer.

8.1 Visuelle Kodierung

EigenschaftSemantik
FarbeGrün = OK · Rot = Exception · Gelb = Slow Call
GrößeAnzahl gleichzeitiger Calls auf dieser Dependency
GeschwindigkeitNormiert auf Antwortzeit — langsamer = längere Dauer
DichteAufruffrequenz — viele Kugeln = hoher Durchsatz
Mehrere gestaffeltParallele Calls (Thread-Parallelität)

8.2 Log-Format

// NDJSON — ein Event pro Zeile
{ "t": 1234567890123, "src": "com.example.ui.TodoController",
  "tgt": "com.example.service.TodoService",
  "method": "getTodos", "durationMs": 42,
  "status": "OK", "threadId": "main" }

8.3 Player-Steuerung

  • Play / Pause — Zeitraffer starten und anhalten
  • Geschwindigkeit — 1× bis 20× (Slider)
  • Scrubbing — beliebige Zeitposition anspringen
  • Filter — nach Paket, Klasse, Status, Zeitfenster
  • Heatmap-Overlay — Straßenbreite dynamisch nach Frequenz anpassen

8.4 Analytische Nutzung

  • Stoßzeiten — dichter Kugelstrom auf bestimmten Pfaden, sofort sichtbar
  • Bottlenecks — langsame Kugeln (gelb) die sich aufstauen
  • Fehlerausbreitung — rote Kugeln breiten sich von einem Ursprung aus
  • Unerwartete Kopplungen — Straßen die unerwartet breit werden
  • Dead Code — Klassen ohne Kugelverkehr über längere Zeit
  • Architekturverletzungen — Fast-Lane-Dependencies die nicht existieren sollten

8.5 Live-Demo

Zufällige Kugeln laufen kontinuierlich auf allen Dependency-Bahnen. Farben kodieren den Call-Typ — in einer echten Implementierung werden sie aus dem Log-Stream gespeist.
